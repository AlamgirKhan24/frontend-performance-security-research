The Two Key Attacks I Studied
XSS
Cross-Site Scripting

Attacker injects malicious JavaScript into a trusted website. Other users' browsers then execute that script unknowingly.

⚠ OWASP Top 10 #3
CSRF
Cross-Site Request Forgery

Attacker tricks an authenticated user's browser into sending an unwanted request — like transferring money or deleting data.

⚠ OWASP Top 10 #1
How These Attacks Work — Step by Step
💉

How XSS Works (Simple Example)

1

An e-commerce website has a comment box or search field that displays user input back on the page without checking it.

2

An attacker types malicious JavaScript code — like <script>document.cookie</script> — into the input field and submits it.

3

The website stores and displays this script. When another user visits the page, their browser runs the attacker's code.

4

The attacker's script can now steal session cookies, redirect the user to a fake login page, or silently send the user's data to the attacker's server.

🔴 Real Impact: XSS can steal login sessions, empty bank accounts, or turn your website into a malware distributor — all without the user realizing anything happened.

🎭

How CSRF Works (Simple Example)

1

A user logs into their bank website. Their browser now holds an active session cookie that proves they are authenticated.

2

The user then opens a spam email and clicks a link that takes them to a malicious website — which has a hidden form on it.

3

That hidden form silently submits a request to the bank's server — for example, to transfer $500 to the attacker's account.

4

The bank's server receives the request. Because it comes from an authenticated browser (with the session cookie), it trusts the request and processes the transfer.

🟣 Real Impact: The user never clicked "Transfer" on the bank site. The attacker forged the action using the victim's own authenticated identity.
